# Phase 1: Effect Substrate

## Goal

Build Handle/Perform in the Rust scheduler: effect bubbling, handler dispatch, continuation management (Resume/Discard/RestartBody), and the stash mechanism for async completions and concurrent effects during suspension.

## Prerequisites

1. **Generational arena migration.** Replace `slab::Slab<Frame>` with a generational arena (e.g. `thunderdome::Arena<Frame>`). FrameId becomes a generational index. Required for stash correctness (see "Frame store" below).

2. **Deferred child dispatch.** Parallel and ForEach currently advance all children synchronously in a loop. If children Perform the same effect, subsequent Performs pile up in the stash — O(n^2) for N children. Refactor both to enqueue children into a pending advance queue, drained by a pump loop at top-level entry points. See `DEFERRED_CHILD_DISPATCH.md`.

## New AST nodes (TypeScript)

```ts
export interface HandleAction {
  kind: "Handle";
  effect_id: number;  // EffectId (opaque u16, max 65535)
  handler: Action;    // handler DAG run when the effect fires
  body: Action;       // body executed within this Handle's scope
}

export interface PerformAction {
  kind: "Perform";
  effect_id: number;  // EffectId (opaque u16, must match some enclosing Handle)
}
```

Each Handle intercepts one effect. Multiple effects in scope = nested Handles. There is no ResumeAction — the handler DAG's tagged output determines what happens to the continuation.

### Continuation operations

Closed enum. Scheduler-internal protocol, not user-extensible.

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum ContinuationOp {
    /// Deliver value to the Perform's parent. Body continues from suspension point.
    Resume,
    /// Tear down the body. Deliver value to the Handle's parent. Handle exits.
    Discard,
    /// Tear down the body. Re-advance the body from scratch with the value.
    RestartBody,
}
```

| Op | Value delivered to | Value type |
|---|---|---|
| Resume | Perform's parent (body continues) | What the body expects at the Perform site |
| Discard | Handle's parent (Handle exits) | The Handle's output type |
| RestartBody | Body re-entry (fresh start) | The body's input type |

All three are `serde_json::Value` on the Rust side. TypeScript types are a discriminated union:

```ts
type StateUpdate<TState> =
  | { kind: "Unchanged" }
  | { kind: "Updated"; value: TState };

type HandlerOutput<TResume, TDiscard, TRestart, TState> =
  | { kind: "Resume"; value: TResume; state_update: StateUpdate<TState> }
  | { kind: "Discard"; value: TDiscard }
  | { kind: "RestartBody"; value: TRestart; state_update: StateUpdate<TState> };
```

Discard has no `state_update` — the frame is being destroyed, so updating its state is pointless. Resume and RestartBody carry state updates because the frame survives.

In practice, no handler produces all three. Each combinator narrows the union:

- ReadVar: Resume only, always Unchanged.
- Throw: Discard only.
- Loop recur: RestartBody only, always Unchanged.
- Loop done: Discard only.
- Retry: RestartBody with Updated(count - 1), or Discard when count hits zero.
- withState Put: Resume with Updated(new_value).

No handler uses both Resume and RestartBody. ReadVar always Resumes. Throw always Discards. Loop uses RestartBody (Continue) or Discard (Break).

Resume and Discard are the canonical two continuation operations in the algebraic effects literature. RestartBody exists because our scheduler is a cooperative slab-based state machine without generalized TCO. Without it, modeling loops as recursion (handler re-invokes Handle) would push a new frame per iteration. RestartBody is a localized trampoline: tear down old body frames, re-advance the body ActionId. O(1) memory.

### Effect routing

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, PartialOrd, Ord)]
pub struct EffectId(pub u16);
```

`u16` to keep `FlatEntry` at 8 bytes. `FlatAction::Handle { effect_id: EffectId, handler: ActionId }` = u16 + u32 = 6 bytes, which fits. A u32 EffectId would break the 8-byte constraint. 65535 effects per workflow is plenty. Generated by the TypeScript builder's monotonic counter. The Rust engine compares integers and never interprets what an effect means.

## Generalized handler state

The Handle frame carries opaque state as `serde_json::Value`. The engine never interprets it. Handler DAGs read and write the state — they're normal AST subgraphs executed by the same engine. Most (readVar, loop control, tryCatch routing) are built entirely from builtins and never leave Rust.

1. When entering a Handle frame, the engine stores its input as `state`. The same input is passed to the body. (If state needs to differ from body input, the TypeScript layer shapes the input via Chain/Parallel prefix.)

2. When a Perform fires, the engine constructs `{ "payload": <perform_value>, "state": <opaque_state> }` and passes it to the handler DAG.

3. The handler DAG produces a tagged output. Resume and RestartBody carry `state_update`. Discard does not.

   ```json
   { "kind": "Resume", "value": <V>, "state_update": { "kind": "Unchanged" } }
   { "kind": "Resume", "value": <V>, "state_update": { "kind": "Updated", "value": <S> } }
   { "kind": "RestartBody", "value": <V>, "state_update": { "kind": "Updated", "value": <S> } }
   { "kind": "Discard", "value": <V> }
   ```

   Rust enum:

   ```rust
   #[derive(Debug, Clone, PartialEq, Eq)]
   pub enum StateUpdate {
       Unchanged,
       Updated(Value),
   }
   ```

Under this architecture, there are no variables and no cache. There is a delimited scope holding an arbitrary state value, and effect handlers that read/write it. `declare` uses state as a read-only dictionary (Reader monad). `withState` uses state as a read/write cell (State monad). `retry` uses state as an integer counter.

## Flat table entries

```rust
pub enum FlatAction<T> {
    // existing: Invoke, Chain, Parallel, ForEach, Branch, Loop, Step

    /// Scoped effect handler. Body starts at action_id + 1 (flattened inline).
    Handle {
        effect_id: EffectId,
        handler: ActionId,
    },

    /// Emit effect. Leaf node (1-entry, like Invoke).
    Perform {
        effect_id: EffectId,
    },
}
```

Handle stores effect_id (u16) and handler (u32) = 6 bytes. The body is not stored in the variant — it starts at `action_id + 1`, flattened inline immediately after the Handle entry. This follows the same pattern as Loop and ForEach: the body position is derivable from the parent's position, so no ChildRef indirection is needed. (ChildRef is only needed for Parallel and ForEach child slots, where N children in a contiguous range may each need to point elsewhere.)

Perform is a 1-entry leaf.

Both pass through `try_map_target` unchanged.

```rust
impl FlatConfig {
    /// Body ActionId for a Handle. Always action_id + 1 (body is flattened inline).
    #[must_use]
    pub fn handle_body(&self, id: ActionId) -> ActionId {
        debug_assert!(matches!(self.action(id), FlatAction::Handle { .. }));
        ActionId(id.0 + 1)
    }
}
```

## Frame store

The frame store must be a generational arena (e.g. `thunderdome::Arena<Frame>`). FrameId is the arena's generational index — an opaque handle containing both a slot index and a generation counter. When a frame is removed, the slot's generation is bumped. If the slot is later reused by a new frame, the old FrameId's generation won't match, and `.get(stale_id)` returns None.

This matters for the stash. A stashed Delivery or Effect holds a ParentRef containing a FrameId. If teardown_body removes the target frame and the slot is reused, the stashed item's FrameId is stale. Without generational indices, `.get(id)` would return the wrong frame. With them, `.get(id)` returns None and the stashed item is dropped.

All frame access in the snippets below uses the generational FrameId directly (`self.frames.get(frame_id)`), not a raw index.

## Frame kind

```rust
#[derive(Debug)]
pub struct HandleFrame {
    pub effect_id: EffectId,
    pub handler: ActionId,
    /// Resolved during advance. Stored for RestartBody re-entry.
    pub body: ActionId,
    /// Opaque state. Initialized from Handle's input.
    /// Mutated by handler output's state_update field.
    pub state: Value,
    /// Suspended continuation while handler is running. None when body is running normally.
    pub continuation: Option<ContinuationRoot>,
}

pub enum FrameKind {
    // existing: Chain, Parallel, ForEach, Loop
    Handle(HandleFrame),
}

/// The suspended continuation: where to deliver on Resume,
/// what to clean up on Discard/RestartBody.
#[derive(Debug)]
struct ContinuationRoot {
    /// The Perform's delivery point. On Resume, deliver here.
    perform_parent: ParentRef,
}
```

At most one continuation per Handle at a time. The body is synchronously frozen at the Perform point, so no further Performs fire from that execution path. If concurrent Performs target the same Handle (e.g. two branches of a Parallel both Perform the same effect), the first dispatches the handler and subsequent ones are stashed until the Handle is free (see "Stash mechanism" below).

### ParentRef::HandlerChild

The handler DAG runs as a child of the Handle frame. Its frames are descendants of the Handle in the slab. When a task inside the handler completes, `find_suspended_ancestor` would walk up and find this Handle's continuation is Some — stashing the handler's own task completion. Deadlock: the handler resolves the suspension, but its completions are stashed against the very suspension it's resolving.

Fix: a new ParentRef variant for handler children. `find_suspended_ancestor` skips a Handle's suspension check when traversed via HandlerChild.

```rust
pub enum ParentRef {
    SingleChild { frame_id: FrameId },
    IndexedChild { frame_id: FrameId, child_index: usize },
    /// Handler DAG child. find_suspended_ancestor skips this Handle's
    /// suspension check when traversed via this variant.
    HandlerChild { frame_id: FrameId },
}
```

`dispatch_to_handler` uses HandlerChild when advancing the handler. The `deliver` method uses HandlerChild to route handler completions directly to `handle_handler_completion` without a runtime check.

## bubble_effect

Walk parent pointers to find a Handle matching the effect ID. If the matching Handle is already suspended (continuation occupied), stash the effect for later.

```rust
fn bubble_effect(
    &mut self,
    starting_parent: Option<ParentRef>,
    effect_id: EffectId,
    payload: Value,
) -> Result<(), AdvanceError> {
    let mut current = starting_parent;

    while let Some(parent_ref) = current {
        let parent_id = parent_ref.frame_id();

        // Frame may have been torn down (stashed effect retried after teardown).
        let Some(parent) = self.frames.get(parent_id) else {
            return Ok(()); // frame tree gone, silently drop
        };

        if let FrameKind::Handle(handle_frame) = &parent.kind {
            if handle_frame.effect_id == effect_id {
                if handle_frame.continuation.is_some() {
                    // Handle is busy (already running a handler for a prior Perform).
                    // Stash this effect. It will be retried after the current handler completes.
                    self.stashed_items.push(StashedItem::Effect {
                        starting_parent,
                        effect_id,
                        payload,
                    });
                    return Ok(());
                }

                let perform_parent = starting_parent
                    .expect("Perform with no parent cannot reach a Handle");

                // Extract handler and state before dropping the immutable borrow.
                let handler = handle_frame.handler;
                let state = handle_frame.state.clone();

                return self.dispatch_to_handler(
                    parent_id, perform_parent, payload, handler, state,
                );
            }
        }

        current = parent.parent;
    }

    Err(AdvanceError::UnhandledEffect { effect_id })
}
```

O(depth of frame tree). Chain, Parallel, Branch, ForEach are invisible — the walk passes through them.

The "Handle is busy" stash replaces the old assert. Two Performs for the same effect in a Parallel are handled naturally: the first dispatches the handler, the second is stashed. When the handler completes and sweep runs, the stashed effect retries and finds the Handle free.

## dispatch_to_handler

```rust
fn dispatch_to_handler(
    &mut self,
    handle_frame_id: FrameId,
    perform_parent: ParentRef,
    payload: Value,
    handler: ActionId,
    state: Value,
) -> Result<(), AdvanceError> {
    // Store the continuation. Re-borrow the Handle frame mutably
    // (bubble_effect's immutable borrow was dropped before this call).
    let handle_frame = self.handle_frame_mut(handle_frame_id);
    handle_frame.continuation = Some(ContinuationRoot { perform_parent });

    // Construct handler input.
    let handler_input = serde_json::json!({
        "payload": payload,
        "state": state,
    });

    // Advance handler as HandlerChild. This prevents find_suspended_ancestor
    // from stashing handler task completions against this Handle's suspension.
    self.advance(
        handler,
        handler_input,
        Some(ParentRef::HandlerChild { frame_id: handle_frame_id }),
    )?;

    Ok(())
}
```

The body's frame tree stays connected — no parent pointers are nulled or rerouted. The Perform point is synchronously stuck (it triggered bubble_effect instead of producing a Dispatch), so no further synchronous execution propagates from that path. The handler runs as a new child of the Handle via HandlerChild.

### Utility: handle_frame_mut

```rust
/// Convenience accessor. Panics if the frame isn't a Handle.
```

## Stash mechanism

Two situations require deferring work:

1. **Async completions during suspension.** A Parallel branch has an in-flight external task. The branch's Handle ancestor is suspended (handler running). When the task completes, delivering its result would advance the frozen body.

2. **Concurrent effects targeting a busy Handle.** Two Performs for the same effect fire in the same Parallel. The first dispatches the handler. The second finds the Handle occupied.

Both cases are handled by a single mechanism: a global stash on WorkflowState. Items are stashed when they can't be processed immediately. After every handler completion, the engine sweeps the stash and retries each item against the current tree state.

This is the [Stash pattern](https://doc.akka.io/docs/akka/current/typed/stash.html) from actor frameworks: when an actor can't process a message in its current state, it stashes it. On state transition, it unstashes and retries.

```rust
#[derive(Debug)]
enum StashedItem {
    /// A task completion waiting to be delivered.
    Delivery {
        parent_ref: ParentRef,
        value: Value,
    },
    /// An effect waiting to be bubbled.
    Effect {
        starting_parent: Option<ParentRef>,
        effect_id: EffectId,
        payload: Value,
    },
}

pub struct WorkflowState {
    // existing fields: frames, flat_config, task_to_parent

    /// Work items deferred because their target Handle was busy or their
    /// target frame was inside a suspended body. Swept after every handler completion.
    stashed_items: Vec<StashedItem>,
}
```

### deliver_or_stash

```rust
fn deliver_or_stash(
    &mut self,
    parent_ref: ParentRef,
    value: Value,
) -> Result<Option<Value>, CompleteError> {
    // Target frame was torn down (Discard/RestartBody removed it).
    if self.frames.get(parent_ref.frame_id()).is_none() {
        return Ok(None);
    }

    if self.find_suspended_ancestor(parent_ref.frame_id()).is_some() {
        self.stashed_items.push(StashedItem::Delivery { parent_ref, value });
        Ok(None)
    } else {
        self.deliver(Some(parent_ref), value)
    }
}
```

### complete_task

```rust
fn complete_task(
    &mut self,
    task_id: TaskId,
    value: Value,
) -> Result<Option<Value>, CompleteError> {
    match self.task_to_parent.remove(&task_id) {
        None => Ok(None), // task's frame was torn down, drop silently
        Some(parent_ref) => self.deliver_or_stash(parent_ref, value),
    }
}
```

### sweep_stash

Called after every handler completion. Takes the entire stash, retries each item.

```rust
fn sweep_stash(&mut self) -> Result<Option<Value>, CompleteError> {
    loop {
        let items = std::mem::take(&mut self.stashed_items);
        if items.is_empty() {
            return Ok(None);
        }
        let count_before = items.len();
        for item in items {
            let result = self.process_stashed_item(item)?;
            if result.is_some() {
                return Ok(result);
            }
        }
        // If nothing made progress (stash size didn't shrink), all items
        // are still blocked. Stop.
        if self.stashed_items.len() >= count_before {
            return Ok(None);
        }
    }
}

fn process_stashed_item(
    &mut self,
    item: StashedItem,
) -> Result<Option<Value>, CompleteError> {
    match item {
        StashedItem::Delivery { parent_ref, value } => {
            self.deliver_or_stash(parent_ref, value)
        }
        StashedItem::Effect { starting_parent, effect_id, payload } => {
            // bubble_effect handles all cases: stashes again if Handle
            // is still busy, dispatches if free, returns UnhandledEffect
            // if the frame tree was torn down.
            self.bubble_effect(starting_parent, effect_id, payload)
                .map_err(CompleteError::from)?;
            Ok(None)
        }
    }
}
```

The loop terminates because each iteration either delivers at least one item (shrinking the stash) or gives up. Delivering a stashed item may Resume a Handle, unblocking further items — the loop catches cascades.

### find_suspended_ancestor

```rust
fn find_suspended_ancestor(&self, frame_id: FrameId) -> Option<FrameId> {
    let mut current = self.frames.get(frame_id)?.parent;
    while let Some(parent_ref) = current {
        let parent_id = parent_ref.frame_id();

        // HandlerChild: skip this Handle's suspension check. Handler
        // deliveries must reach the Handle to resolve the suspension.
        if !matches!(parent_ref, ParentRef::HandlerChild { .. }) {
            if let FrameKind::Handle(handle_frame) = &self.frames[parent_id].kind {
                if handle_frame.continuation.is_some() {
                    return Some(parent_id);
                }
            }
        }

        current = self.frames.get(parent_id)?.parent;
    }
    None
}
```

The HandlerChild check prevents deadlock. Without it, a multi-step handler like `Chain(Invoke(step1), Invoke(step2))` would have its intermediate task completions stashed against the Handle's own suspension. The handler resolves the suspension, so stashing its completions creates a cycle.

### Cancelled tasks

When `teardown_body` runs (Discard or RestartBody), it removes body frames and their `task_to_parent` entries. Late-arriving completions find no entry and are silently dropped. Stashed items targeting torn-down frames are dropped during the next sweep. No explicit cancel signal is sent to the external driver.

### Example

```
Handle(effect, handler,
  Parallel(
    Chain(Invoke(taskA), Perform(effect)),
    Chain(Invoke(taskB), Perform(effect)),
  )
)
```

1. Advance: Parallel dispatches both branches. Invoke A and B dispatch external tasks.
2. Task A completes. deliver -> Chain A trampolines -> Perform -> bubble_effect -> Handle is free -> dispatch_to_handler. Continuation stored, handler starts.
3. Task B completes. deliver_or_stash -> find_suspended_ancestor finds Handle suspended -> **stashed as Delivery**.
4a. Handler Resumes: deliver to Parallel slot 0. sweep_stash -> deliver B -> Chain B trampolines -> Perform -> bubble_effect -> Handle is free -> dispatch_to_handler. Second handler starts. Second handler Resumes -> Parallel joins -> Handle exits.
4b. Handler Discards: teardown_body removes Parallel, Chain B, task entries. sweep_stash -> stashed B's target frame is gone -> **dropped**. Handle exits with discard value.

### Example: concurrent Performs

```
Handle(effect, handler,
  Parallel(
    Perform(effect),
    Perform(effect),
  )
)
```

1. Advance: Parallel advances both children.
2. First Perform -> bubble_effect -> Handle is free -> dispatch_to_handler. Continuation stored.
3. Second Perform -> bubble_effect -> Handle has continuation -> **stashed as Effect**.
4. Handler Resumes: deliver to Parallel slot 0. sweep_stash -> retries stashed effect -> bubble_effect -> Handle is free -> dispatch_to_handler. Second handler starts.
5. Second handler Resumes -> Parallel slot 1 -> joins -> Handle exits.

## Handler completion

The `deliver` method routes on ParentRef:

```rust
ParentRef::HandlerChild { frame_id } => {
    // Handler DAG completed.
    return self.handle_handler_completion(frame_id, value);
}

ParentRef::SingleChild { .. } => {
    let frame = self.frames.remove(frame_id);
    match frame.kind {
        FrameKind::Chain { rest } => {
            self.advance(rest, value, frame.parent)?;
            Ok(None)
        }
        FrameKind::Loop { body } => {
            // ... existing Loop logic ...
        }
        FrameKind::Handle(_) => {
            // Body completed without Performing. Handle exits normally.
            self.deliver(frame.parent, value)
        }
        _ => unreachable!("SingleChild: Chain, Loop, or Handle"),
    }
}
```

HandlerChild distinguishes handler completion from body completion structurally — no runtime check on `continuation.is_some()`.

### handle_handler_completion

```rust
fn handle_handler_completion(
    &mut self,
    handle_frame_id: FrameId,
    output: Value,
) -> Result<Option<Value>, CompleteError> {
    let kind_str = output["kind"].as_str()
        .ok_or_else(|| CompleteError::InvalidHandlerOutput {
            value: output.clone(),
        })?;
    let value = output["value"].clone();

    let handle_frame = self.handle_frame_mut(handle_frame_id);
    let continuation = handle_frame.continuation.take()
        .expect("handler completion requires stored continuation");

    let result = match kind_str {
        "Resume" => {
            self.apply_state_update(handle_frame_id, &output)?;
            self.resume_continuation(handle_frame_id, continuation, value)
        }
        "Discard" => {
            self.discard_continuation(handle_frame_id, continuation, value)
        }
        "RestartBody" => {
            self.apply_state_update(handle_frame_id, &output)?;
            self.restart_body(handle_frame_id, continuation, value)
        }
        _ => Err(CompleteError::InvalidHandlerOutput { value: output }),
    }?;

    // After any continuation op, sweep the stash. Items blocked by this
    // Handle's suspension may now be deliverable. Items whose frames were
    // torn down (Discard/RestartBody) are silently dropped.
    if result.is_some() {
        return Ok(result);
    }
    self.sweep_stash()
}

fn apply_state_update(
    &mut self,
    handle_frame_id: FrameId,
    output: &Value,
) -> Result<(), CompleteError> {
    let state_update_kind = output["state_update"]["kind"].as_str()
        .ok_or_else(|| CompleteError::InvalidHandlerOutput {
            value: output.clone(),
        })?;
    match state_update_kind {
        "Unchanged" => Ok(()),
        "Updated" => {
            let new_state = output["state_update"]["value"].clone();
            let handle_frame = self.handle_frame_mut(handle_frame_id);
            handle_frame.state = new_state;
            Ok(())
        }
        _ => Err(CompleteError::InvalidHandlerOutput { value: output.clone() }),
    }
}
```

## Resume

Deliver the value to where the Perform would have delivered. The body subgraph is intact (never severed), so this is identical to what would have happened if the Perform had been a normal Invoke that completed.

```rust
fn resume_continuation(
    &mut self,
    _handle_frame_id: FrameId,
    continuation: ContinuationRoot,
    value: Value,
) -> Result<Option<Value>, CompleteError> {
    self.deliver(Some(continuation.perform_parent), value)
}
```

sweep_stash runs after this returns (in handle_handler_completion), picking up deliveries and effects unblocked by this Handle's un-suspension.

## Discard

Tear down the body. Handle exits with the value.

```rust
fn discard_continuation(
    &mut self,
    handle_frame_id: FrameId,
    _continuation: ContinuationRoot,
    value: Value,
) -> Result<Option<Value>, CompleteError> {
    self.teardown_body(handle_frame_id);
    let parent = self.frames[handle_frame_id].parent;
    self.frames.remove(handle_frame_id);
    Ok(self.deliver(parent, value)?)
}
```

## RestartBody

Tear down the body. Re-advance from scratch.

```rust
fn restart_body(
    &mut self,
    handle_frame_id: FrameId,
    _continuation: ContinuationRoot,
    value: Value,
) -> Result<Option<Value>, CompleteError> {
    self.teardown_body(handle_frame_id);
    let handle_frame = self.handle_frame_mut(handle_frame_id);
    let body = handle_frame.body;
    self.advance(body, value, Some(ParentRef::SingleChild { frame_id: handle_frame_id }))?;
    Ok(None)
}
```

## teardown_body

Remove all descendant frames and their task_to_parent entries. In-flight tasks run to completion; their results are silently dropped when complete_task finds no entry.

```rust
fn teardown_body(&mut self, handle_frame_id: FrameId) {
    let body_frames: Vec<FrameId> = self.frames
        .iter()
        .filter_map(|(frame_id, _)| {
            if self.is_descendant_of(frame_id, handle_frame_id) {
                Some(frame_id)
            } else {
                None
            }
        })
        .collect();

    for frame_id in &body_frames {
        self.frames.remove(frame_id);
    }

    let body_frame_set: std::collections::HashSet<FrameId> =
        body_frames.into_iter().collect();
    self.task_to_parent.retain(|_, parent| {
        match parent {
            Some(parent_ref) => !body_frame_set.contains(&parent_ref.frame_id()),
            None => true,
        }
    });
}

fn is_descendant_of(&self, frame_id: FrameId, ancestor_id: FrameId) -> bool {
    let mut current = self.frames.get(frame_id).and_then(|f| f.parent);
    while let Some(parent_ref) = current {
        let parent_id = parent_ref.frame_id();
        if parent_id == ancestor_id {
            return true;
        }
        current = self.frames.get(parent_id).and_then(|f| f.parent);
    }
    false
}
```

O(n * depth) where n is the slab size. Acceptable for Phase 1. Can optimize later with child-pointer tracking or generation counters.

## Flattener changes

### Tree AST

```rust
pub enum Action {
    // existing variants...
    Handle(HandleAction),
    Perform(PerformAction),
}

pub struct HandleAction {
    pub effect_id: EffectId,
    pub handler: Box<Action>,
    pub body: Box<Action>,
}

pub struct PerformAction {
    pub effect_id: EffectId,
}
```

### flatten_action_at

```rust
Action::Handle(HandleAction { effect_id, handler, body }) => {
    // Body flattened inline at action_id + 1. No child slot, no ChildRef.
    // flatten_action calls alloc(), which appends at the next position,
    // so body_id is guaranteed to be action_id + 1.
    let body_id = self.flatten_action(*body, workflow_root)?;
    debug_assert_eq!(body_id.0, action_id.0 + 1);
    let handler_id = self.flatten_action(*handler, workflow_root)?;
    FlatAction::Handle { effect_id, handler: handler_id }
}

Action::Perform(PerformAction { effect_id }) => {
    FlatAction::Perform { effect_id }
}
```

The body starts at `action_id + 1` because `flatten_action` immediately allocates the next position. The handler is flattened after the entire body subtree. This follows the same pattern as Loop and ForEach: body position is derivable, no child slot or ChildRef needed.

### fill_child_slot

Handle is multi-entry (Handle entry + inline body subtree), so when a Handle appears as the child of another action (e.g. Chain's `first`), it gets a ChildRef:

```rust
fn fill_child_slot(&mut self, action: Action, ...) {
    match action {
        Action::Chain { .. } | Action::Parallel { .. } | Action::Branch { .. }
        | Action::Handle { .. } => {
            let action_id = self.flatten_action(action, workflow_root)?;
            self.entries[slot.0 as usize] = Some(FlatEntry::ChildRef { action: action_id });
        }
        single_entry => { /* inline */ }
    }
}
```

### try_map_target

```rust
FlatAction::Handle { effect_id, handler } => FlatAction::Handle { effect_id, handler },
FlatAction::Perform { effect_id } => FlatAction::Perform { effect_id },
```

Both pass through unchanged.

## Interaction with existing frame kinds

Chain, Parallel, Branch, ForEach: unchanged. If a child suspends via bubble_effect, the parent frame sits dormant with a pending slot. When resumed, the child delivers and the parent proceeds. This is structurally identical to waiting for an external Invoke to complete.

Loop (before Phase 4 migration): still exists as a separate frame kind. Doesn't interact with Handle/Perform. Replaced by nested Handles in Phase 4.
