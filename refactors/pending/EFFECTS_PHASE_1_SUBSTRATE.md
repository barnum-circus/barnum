# Phase 1: Effect Substrate

## Goal

Build the structural routing mechanism for algebraic effects in the Rust scheduler. After this phase, a Perform can suspend execution, bubble up to a Handle, and be resumed or discarded. No semantic effects yet — just the plumbing.

## Prerequisites

None. This is the foundation.

## New AST nodes (tree AST, TypeScript)

```ts
export interface HandleAction {
  kind: "Handle";
  effect: EffectId;   // the one effect this Handle intercepts
  handler: Action;    // the handler DAG to run when the effect fires
  body: Action;       // the body to execute within this Handle's scope
}

export interface PerformAction {
  kind: "Perform";
  effect: EffectId;   // must match some enclosing Handle's effect
}
```

Two new nodes. Each Handle intercepts exactly one effect. If you need to handle multiple effects, nest multiple Handles — that's composition.

There is no `ResumeAction`. Resumption, discarding, and body re-entry are the Handle frame's interpretation of the handler DAG's tagged output:

```ts
// Handler DAG output — one of three continuation operations:
| { kind: "Resume", value: Value }       // deliver value to suspended continuation
| { kind: "Discard", value: Value }      // tear down continuation, exit Handle with value
| { kind: "RestartBody", value: Value }  // tear down continuation, re-enter body with value
```

### Effect routing: opaque gensym'd IDs

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct EffectId(pub u32);
```

Opaque `u32` generated by the TypeScript builder's monotonic counter. The Rust engine compares integers. It never interprets what an effect means. No enum, no strings, no Rust changes for new effects.

## New frame kind (Rust scheduler)

```rust
pub enum FrameKind {
    // ... existing variants ...

    Handle {
        effect: EffectId,
        handler: ActionId,
        /// The suspended continuation, if a Perform has fired and the handler is running.
        /// None when the body is running normally.
        /// Some when the body is suspended and the handler is in flight.
        continuation: Option<ContinuationRoot>,
    },
}

struct ContinuationRoot {
    /// The severed frame (root of the disconnected subgraph).
    frame_id: FrameId,
    /// Where it was attached before we severed it (for reconnection on Resume).
    original_parent: ParentRef,
}
```

One Handle, one effect, one handler, one continuation. All singular. At most one continuation exists at a time because the entire body subgraph is suspended when a Perform fires — nothing else in the body can Perform while it's suspended.

## bubble_effect: the core traversal

When a Perform is evaluated, the scheduler walks up parent pointers looking for a Handle frame whose `effect` field matches.

```rust
fn bubble_effect(
    &mut self,
    starting_frame: FrameId,
    effect: EffectId,
    payload: Value,
) -> Result<(), WorkflowError> {
    let mut current = self.frames[starting_frame].parent;

    while let Some(parent_ref) = current {
        let parent_id = parent_ref.frame_id();
        let parent = &self.frames[parent_id];

        if let FrameKind::Handle { effect: handled_effect, .. } = &parent.kind {
            if *handled_effect == effect {
                return self.dispatch_to_handler(parent_id, payload);
            }
        }

        current = parent.parent;
    }

    Err(WorkflowError::UnhandledEffect(effect))
}
```

That's it. Walk up, compare integers, stop on match. Chain, Parallel, Branch, ForEach are invisible — the traversal walks right past them.

## Continuation lifecycle

Three things can happen, and they're all simple.

### 1. Perform fires → body suspends, handler runs

```
Before:                         After:

Handle ──► body subgraph        Handle ──► handler DAG
                                    │
                                    └─ continuation (severed, stored in Handle.continuation)
```

The scheduler severs the body subgraph from the Handle frame and stores it as the continuation. Then it creates a new frame for the handler DAG and advances it with the Perform's payload.

### 2. Handler completes → tagged output determines what happens

The handler DAG produces a value. The Handle frame reads the `kind` field:

**Resume**: Reconnect the continuation. Deliver the value into it. Body wakes up and continues from where it left off.

```
Handle ──► body subgraph (reconnected, running again)
```

**Discard**: Tear down the continuation immediately (remove all its frames from the slab, cancel pending external tasks). Deliver the value to the Handle frame's parent. The Handle is done.

```
Handle exits, delivers value to parent
(continuation frames removed from slab)
```

**RestartBody**: Tear down the continuation immediately. Re-advance the body from scratch with the value.

```
Handle ──► body subgraph (fresh, re-entered with new value)
(old continuation frames removed from slab)
```

### 3. Body completes normally → Handle exits

If the body completes without ever Performing (or after all Performs were Resumed), the Handle frame delivers the body's result to its parent. No continuation to clean up — `continuation` is `None`.

## Interaction with existing frame kinds

### Chain

No changes. If a child suspends (via bubble_effect walking past the Chain frame), the Chain frame sits dormant. When the continuation is resumed and the child eventually delivers a value, the Chain frame's existing `deliver` logic advances to the rest.

### Parallel

No changes. If child A suspends, A's slot in the Parallel frame's results vector remains empty. When the continuation is resumed and A delivers, the Parallel frame's existing logic checks if all slots are full and joins if so. This is structurally identical to waiting for an external Invoke to complete.

### Branch

No changes. Same pattern as Chain.

### ForEach

No changes. Same pattern as Parallel — each iteration is a slot, suspended iterations are empty slots.

### Loop (before Phase 4 migration)

During Phase 1, Loop still exists as a separate frame kind. It doesn't interact with Handle/Perform. After Phase 4, Loop is replaced by Handle(LoopControl).

## Test strategy

### Unit tests (Rust)

1. **Perform without Handle**: Verify `UnhandledEffect` error.
2. **Resume tagged output**: Handler DAG produces `{ kind: "Resume", value }`. Verify the continuation receives the value and the body completes.
3. **Discard tagged output**: Handler DAG produces `{ kind: "Discard", value }`. Verify the continuation is torn down and the Handle frame exits with the value.
4. **RestartBody tagged output**: Handler DAG produces `{ kind: "RestartBody", value }`. Verify the old continuation is torn down and the body is re-entered with the value.
5. **Nested Handle**: Inner Handle intercepts one effect, outer Handle intercepts another. Verify correct routing.
6. **Handle + Perform in Chain**: Perform is the first half of a Chain. After Resume, the rest of the Chain executes.
7. **Handle + Perform in Parallel**: One parallel branch performs an effect. The other completes normally. After Resume, the Parallel joins both results.
8. **Discard cleans up slab**: After Discard, verify all continuation frames are removed from the slab and pending external tasks are cancelled.
9. **Multiple Performs (sequential)**: Body performs multiple effects in a Chain. Each is handled and resumed. Verify all resume correctly.
10. **RestartBody loop pattern**: Handler always produces RestartBody until a condition is met, then Discard. Verify multiple re-entries and final exit. (This validates the loop pattern without LoopControl semantics.)
11. **Perform across ForEach**: An action inside ForEach performs an effect. Verify the ForEach iteration suspends and resumes correctly.

### Integration tests

1. **End-to-end with TypeScript handler**: The handler DAG includes an Invoke that calls a TypeScript function. The function's result feeds into a tagged output. The Handle frame interprets it and resumes the continuation.
2. **Nested handlers with TypeScript**: Multiple Handle blocks, TypeScript handlers at different levels.

## Deliverables

1. `EffectId(u32)` type
2. `FlatAction::Handle` and `FlatAction::Perform` in the flat table
3. `FrameKind::Handle` in the scheduler
4. `bubble_effect` traversal (walk up, compare integers)
5. Tagged output interpretation: Resume, Discard, RestartBody
6. `teardown_continuation` (depth-first frame removal + task cancellation)
7. Flattener support (tree AST Handle/Perform → flat table entries)
8. Tests per above

## What this phase does NOT include

- No semantic effects (ReadVar, Throw, LoopControl). Those come in Phases 2-4.
- No TypeScript surface API changes. Declare, tryCatch, loop rewriting come later.
- No RAII / Bracket. That's Phase 5.
- No durable suspension. That's Phase 6.

This phase builds infrastructure only. The test suite validates the mechanism using synthetic effects (handler DAGs that produce each of the three tagged outputs).

## Note on cycle hazards

The frame tree is always acyclic. StepAction (and RestartBody) create new frames per invocation — they never create pointers back to existing frames. `teardown_continuation` is a simple depth-first traversal of a tree, not a graph traversal. No cycle detection is needed.
