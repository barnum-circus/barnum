# Phase 1: Effect Substrate

## Goal

Build the structural routing mechanism for algebraic effects in the Rust scheduler. After this phase, a Perform can suspend execution, bubble up to a Handle, and be resumed or discarded. No semantic effects yet — just the plumbing.

## Prerequisites

None. This is the foundation.

## New AST nodes (tree AST, TypeScript)

```ts
export interface HandleAction {
  kind: "Handle";
  handlers: Record<string, Action>;  // effect_type -> handler DAG
  body: Action;
}

export interface PerformAction {
  kind: "Perform";
  effect: string;  // routing key (matches a key in HandleAction.handlers)
}
```

Two new nodes, not three. There is no `ResumeAction`. Resumption, discarding, and body re-entry are NOT AST nodes — they are interpretations of the handler DAG's output by the Handle frame.

Handle maps effect type strings to handler DAGs. When a Perform fires with a matching effect type, the handler DAG is advanced with `{ payload: <pipeline_value> }`. The handler DAG produces a **tagged output** that the Handle frame interprets:

```ts
// Handler DAG output — one of three continuation operations:
| { kind: "Resume", value: Value }       // deliver value to suspended continuation
| { kind: "Discard", value: Value }      // drop continuation, exit Handle with value
| { kind: "RestartBody", value: Value }  // drop continuation, re-enter body with value
```

The Handle frame is a pure structural router. It doesn't know what effects mean. It knows three things: resume, discard, restart. All semantic meaning lives in the handler DAGs.

### Effect routing: opaque gensym'd IDs

Effect types are opaque `u32` IDs generated by the TypeScript builder's monotonic counter (the same gensym mechanism used for declare IDs and step references). The Rust engine routes on these IDs without interpreting them.

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct EffectId(pub u32);
```

No enum, no strings. The Rust scheduler matches integers. New effects never require Rust changes. Collision is impossible by construction. See EFFECTS_ROADMAP.md for the full rationale.

## New flat table entries (Rust)

```rust
pub enum FlatAction {
    // ... existing variants ...

    /// Install a scoped effect handler.
    Handle {
        /// Maps effect type to the ActionId of the handler DAG.
        handlers: BTreeMap<EffectType, ActionId>,
        /// The body to execute within the handler's scope.
        body: ActionId,
    },

    /// Emit an effect. Pipeline value becomes the payload.
    Perform {
        effect: EffectType,
    },
}
```

No `FlatAction::Resume`. Resumption is handled internally by the Handle frame when it interprets the handler DAG's tagged output.

## New frame kind (Rust scheduler)

```rust
pub enum FrameKind {
    // ... existing variants ...

    /// Scoped effect handler. Intercepts matching effects from the body.
    Handle {
        /// Maps effect type to handler ActionId.
        handlers: BTreeMap<EffectType, ActionId>,
        /// Active handler invocations. Each maps the handler's frame ID
        /// back to the continuation root it was dispatched for.
        active_handlers: HashMap<FrameId, ContinuationRoot>,
        /// Continuations awaiting cleanup (handler completed with Discard/RestartBody).
        orphaned_continuations: Vec<FrameId>,
    },
}

struct ContinuationRoot {
    /// The severed frame (root of the disconnected subgraph).
    frame_id: FrameId,
    /// The original parent ref (for reconnection on Resume).
    original_parent: ParentRef,
}
```

The Handle frame stores the mapping from active handler frames to their associated continuations. No cont_id is exposed to handler DAGs.

When a Perform bubbles up to this frame, it:
1. Severs the parent link (creating a disconnected subgraph = the continuation)
2. Stores the continuation root internally, keyed by the handler invocation's frame
3. Advances the handler DAG with `{ payload }` — just the payload, nothing else

When a handler DAG completes, the Handle frame reads its tagged output and acts:
- `{ kind: "Resume", value }`: Reconnect the continuation's parent link. Deliver `value` into the continuation's root frame. Remove from active_handlers.
- `{ kind: "Discard", value }`: Move the continuation to orphaned_continuations. Deliver `value` to the Handle frame's parent (exiting the Handle scope).
- `{ kind: "RestartBody", value }`: Move the continuation to orphaned_continuations. Re-advance the Handle's body with `value`.

When the Handle frame exits (body completes or Discard triggers exit):
1. All orphaned continuations are torn down (frames removed from slab, pending tasks cancelled)
2. The Handle frame delivers the result to its parent

## bubble_effect: the core traversal

```rust
impl WorkflowState {
    /// Walk parent pointers to find a Handle frame matching the effect.
    /// Intermediate nodes (Chain, Parallel, Branch, ForEach) are skipped.
    fn bubble_effect(
        &mut self,
        starting_frame: FrameId,
        effect: EffectType,
        payload: Value,
    ) -> Result<(), WorkflowError> {
        let mut current = self.frames[starting_frame].parent;
        let mut child_frame_id = starting_frame;

        while let Some(parent_ref) = current {
            let parent_id = parent_ref.frame_id();
            let parent = &self.frames[parent_id];

            if let FrameKind::Handle { handlers, .. } = &parent.kind {
                if handlers.contains_key(&effect) {
                    // Found matching handler.
                    // 1. Sever parent link of child_frame_id.
                    // 2. Store continuation in Handle frame.
                    // 3. Advance handler DAG with { payload, cont_id }.
                    return self.dispatch_to_handler(parent_id, child_frame_id, effect, payload);
                }
            }

            // Not handled here — keep walking up.
            child_frame_id = parent_id;
            current = parent.parent;
        }

        Err(WorkflowError::UnhandledEffect(effect))
    }
}
```

Key properties:
- Chain, Parallel, Branch, ForEach are completely unaware of effects. They sit dormant in the slab.
- The traversal is O(depth) where depth is the number of frames between Perform and Handle. In practice, this is small (< 20).
- If no Handle matches, it's an error. This is analogous to an unhandled exception.

## Continuation lifecycle

### Creation (on Perform)

1. `advance` evaluates a Perform node. Instead of returning a value, it calls `bubble_effect`.
2. `bubble_effect` walks up to the Handle frame.
3. The Handle frame severs the parent link of the frame immediately below it (the continuation root).
4. It stores the continuation root in `active_handlers`, keyed by the handler invocation's frame ID.
5. It advances the handler DAG with `{ payload }`.

### Handler completion → tagged output interpretation

When a handler DAG delivers its result to the Handle frame, the Handle frame reads the `kind` tag:

1. **Resume** (`{ kind: "Resume", value }`):
   - Look up the continuation root associated with this handler invocation.
   - Reconnect the continuation root's parent link to the Handle frame.
   - Call `deliver` on the continuation root with `value`.
   - The dormant subgraph wakes up and execution proceeds normally.
   - Remove the entry from `active_handlers`.

2. **Discard** (`{ kind: "Discard", value }`):
   - Move the continuation root to `orphaned_continuations`.
   - Deliver `value` to the Handle frame's parent (exiting the Handle scope).
   - Remove the entry from `active_handlers`.

3. **RestartBody** (`{ kind: "RestartBody", value }`):
   - Move the continuation root to `orphaned_continuations`.
   - Re-advance the Handle frame's body with `value`.
   - Remove the entry from `active_handlers`.

### Cleanup (on Handle exit)

1. The Handle frame exits (body completes, or Discard triggers exit).
2. All orphaned continuations are torn down:
   - Traverse downward from each root frame.
   - For Parallel/ForEach frames: recurse into children.
   - Cancel pending external tasks.
   - Remove all frames from the slab.
3. Any still-active handler invocations are also torn down (their continuations and handler frames).
4. The Handle frame delivers the result to its parent.

## Interaction with existing frame kinds

### Chain

No changes. If a child suspends (via bubble_effect walking past the Chain frame), the Chain frame sits dormant. When the continuation is resumed and the child eventually delivers a value, the Chain frame's existing `deliver` logic advances to the rest.

### Parallel

No changes. If child A suspends, A's slot in the Parallel frame's results vector remains empty. When the continuation is resumed and A delivers, the Parallel frame's existing logic checks if all slots are full and joins if so. This is structurally identical to waiting for an external Invoke to complete.

### Branch

No changes. Same pattern as Chain.

### ForEach

No changes. Same pattern as Parallel — each iteration is a slot, suspended iterations are empty slots.

### Loop (before Phase 4 migration)

During Phase 1, Loop still exists as a separate frame kind. It doesn't interact with Handle/Perform. After Phase 4, Loop is replaced by Handle(LoopControl).

## Test strategy

### Unit tests (Rust)

1. **Perform without Handle**: Verify `UnhandledEffect` error.
2. **Resume tagged output**: Handler DAG produces `{ kind: "Resume", value }`. Verify the continuation receives the value and the body completes.
3. **Discard tagged output**: Handler DAG produces `{ kind: "Discard", value }`. Verify the continuation is torn down and the Handle frame exits with the value.
4. **RestartBody tagged output**: Handler DAG produces `{ kind: "RestartBody", value }`. Verify the old continuation is torn down and the body is re-entered with the value.
5. **Nested Handle**: Inner Handle intercepts one effect, outer Handle intercepts another. Verify correct routing.
6. **Handle + Perform in Chain**: Perform is the first half of a Chain. After Resume, the rest of the Chain executes.
7. **Handle + Perform in Parallel**: One parallel branch performs an effect. The other completes normally. After Resume, the Parallel joins both results.
8. **Discard cleans up slab**: After Discard, verify all continuation frames are removed from the slab and pending external tasks are cancelled.
9. **Multiple Performs (sequential)**: Body performs multiple effects in a Chain. Each is handled and resumed. Verify all resume correctly.
10. **RestartBody loop pattern**: Handler always produces RestartBody until a condition is met, then Discard. Verify multiple re-entries and final exit. (This validates the loop pattern without LoopControl semantics.)
11. **Perform across ForEach**: An action inside ForEach performs an effect. Verify the ForEach iteration suspends and resumes correctly.

### Integration tests

1. **End-to-end with TypeScript handler**: The handler DAG includes an Invoke that calls a TypeScript function. The function's result feeds into a tagged output. The Handle frame interprets it and resumes the continuation.
2. **Nested handlers with TypeScript**: Multiple Handle blocks, TypeScript handlers at different levels.

## Deliverables

1. `EffectType` enum (or string, per decision above)
2. `FlatAction::Handle` and `FlatAction::Perform` in the flat table
3. `FrameKind::Handle` in the scheduler (with tagged output interpretation)
4. `bubble_effect` traversal
5. Tagged output interpretation: Resume, Discard, RestartBody
6. Slab cleanup on discard / restart
7. Flattener support (tree AST Handle/Perform → flat table entries)
8. Tests per above

## What this phase does NOT include

- No semantic effects (ReadVar, Throw, LoopControl). Those come in Phases 2-4.
- No TypeScript surface API changes. Declare, tryCatch, loop rewriting come later.
- No RAII / Bracket. That's Phase 5.
- No durable suspension. That's Phase 6.

This phase builds infrastructure only. The test suite validates the mechanism using synthetic effects (handler DAGs that produce each of the three tagged outputs).

## Note on cycle hazards

The frame tree is always acyclic. StepAction (and RestartBody) create new frames per invocation — they never create pointers back to existing frames. `teardown_continuation` is a simple depth-first traversal of a tree, not a graph traversal. No cycle detection is needed.
